# 01 AI Execution Rules

> 微购 Design System Skill / 04-ai-rules  
> Version 2.0  
> 本文档定义 AI 执行微购设计任务的总流程。

---

# 1. 执行目标

AI 的目标不是自由创作界面，而是基于微购设计系统输出稳定、可执行、符合规范的结果。

AI 必须保证：

- 不偏离微购设计原则
- 不绕过 Token 体系
- 不自创组件规范
- 不输出空泛建议
- 不制造视觉噪音

---

# 2. 标准执行流程

AI 执行任何界面任务时，按 4 阶段闭环执行。每阶段有明确门禁，未通过不得进入下一阶段。

```text
阶段一：需求确认（唯一用户门禁）
  执行 08-requirement-confirmation-rules.md
  确认用户任务 / 业务目标 / 使用场景
  ↓ 门禁：用户确认"对吗？"
↓
阶段二：设计决策（内部门禁，AI 自主，产物驱动）
  2.1 页面目标与类型 → 2.2 信息层级 → 2.3 布局模式
  → 2.4 组件匹配 → 2.5 Token 类别 / 2.6 交互链路
  每步产出内部产物，依赖链强制执行
  ↓ 门禁：6 步产物全部完成
↓
阶段三：实现
  前置准备（tokens.css / 字体 / iconfont / components.css）
  主体实现（HTML / app.css / app.js）
  ↓ 门禁：项目结构完整、资源路径有效、样式加载顺序正确
↓
阶段四：验证与输出
  浏览器验证 + 设计产物落地验证 + 合规验证
  ↓ 门禁：自查清单逐项通过
  输出项目目录 + 简短运行说明
```

阶段和门禁的详细定义以 `SKILL.md` 为准。本文档不重复具体设计规则，只说明执行流程的总体结构。

---

# 3. 输入判断

任务类型和读取路由以 `SKILL.md` 为准。本文档只说明进入对应任务后的处理方式：

| 用户需求 | 执行方式 |
|---|---|
| 生成完整页面或单个组件 | 执行 UI Generation Rules |
| 审查已有界面 | 执行 UI Review Rules |
| 优化已有界面 | 先审查，再执行对应的生成规则 |
| 检查 Token 合规性 | 执行 Token Usage Rules |
| 询问规范 | 直接回答当前路由选中的规则 |

“直接输出”“不要解释”等要求只影响表达方式，不产生新的任务路由。

---

# 4. 调用边界

本文档只提供执行流程细则，不定义文档读取顺序。

任务分类、必读文件、禁止读取目录和输出形式统一由 `SKILL.md` 决定。不得因为本文档或其他详细规则提到某个目录，就扩大当前任务的读取范围。

当前产品设计阶段只按 `SKILL.md` 路由读取 `03-components/registry.json`、公共规则和命中的组件文件。组件规范缺失时使用页面级临时结构完成原型链路并按缺失内容格式标注，不得使用 KuiklyUI 文档补齐 Web 原型规则。

---

# 5. 缺失内容处理

缺失 Token 时：

```text
当前使用：
原因：
建议新增：
归属位置：
```

缺失组件规范时：

```text
当前使用：
缺失内容：
建议补充：
归属位置：
```

缺失页面信息时：

```text
当前假设：
影响范围：
建议确认：
```

---

# 6. 输出要求

AI 输出必须满足：

- 结论明确
- 层级清晰
- 规则可追溯
- 建议可执行
- 代码可落地
- 文档可直接保存

不允许输出：

```text
可以更高级
视觉更现代
建议更美观
可以更有设计感
```

必须说明：

```text
哪里有问题
为什么有问题
怎么修改
依据什么规则
```

---

# 7. 简短模式

当用户要求“直接输出”“不要解释”“给我文件”时：

只输出最终结果。

不输出过程分析。

---

# 8. 最终原则

AI 始终按微购设计系统工作，而不是按通用 UI 审美自由发挥。
