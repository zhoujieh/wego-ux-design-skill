# 03 UI Review Rules

> 微购 Design System Skill / 04-ai-rules  
> Version 2.0  
> 本文档定义 AI 审查微购界面时的流程和标准。

---

# 1. 审查目标

AI 审查界面时，不只判断好不好看，而是判断是否符合：

- 页面目标
- 信息层级
- 微购设计原则
- Token 规范
- 当前阶段可用的 HTML 组件规则
- 状态完整性
- 可落地性

---

# 2. 审查顺序

```text
页面目标
↓
信息结构
↓
视觉层级
↓
布局节奏
↓
组件选择
↓
Token 使用
↓
状态完整性
↓
问题优先级
↓
修改建议
```

---

# 3. 页面目标审查

检查：

- 页面目标是否明确
- 用户下一步是否清晰
- 主操作是否突出
- 是否存在多个冲突目标
- 是否有无关信息干扰

页面目标不清晰时，优先指出。

---

# 4. 信息层级审查

检查：

- 是否只有一个核心视觉重心
- 标题、正文、辅助说明是否层级清晰
- 强信息是否过多
- 弱信息是否抢占注意力
- 状态信息是否靠近对应对象

---

# 5. 布局审查

检查：

- 是否使用 M0 / M1 / M2 / M3
- 是否正确使用 G1 / G2
- 页面边距是否稳定
- 模块间距是否表达正确关系
- 是否通过临时 padding 伪造布局

---

# 6. Token 审查

检查：

- 是否使用 `wg.*`
- 业务样式是否直接写 HEX / RGBA
- 业务样式是否直接写 px 魔法值
- 是否使用未定义字号
- 是否使用未定义间距
- 是否使用未定义圆角
- 是否使用随机 z-index

---

# 7. 组件审查

先读取 `03-components/registry.json`。界面中出现已注册组件时，再读取 `shared-rules.md` 和对应组件文件；未注册结构按页面实现审查其语义、交互、状态和可访问性，并标注为组件候选，不因组件尚未注册而判定原型不能实现。

不得扫描整个组件目录，也不得使用 KuiklyUI API 作为审查依据。

检查：

- 是否选择了正确组件
- 是否先按组件语义维度选择变体
- 是否遵守组件允许组合和 Anatomy
- 是否出现多个主按钮
- 是否用弹窗承载过长流程
- 是否用 Toast 承载关键决策
- 是否为单个页面创造组件变体
- 是否只补齐组件声明的适用状态

---

# 8. 状态审查

检查：

```text
是否覆盖当前组件规则声明的状态
是否使用原生属性或 ARIA 表达状态
是否错误添加组件未定义的状态
未定义状态是否标注规范缺失
```

危险操作检查：

```text
是否需要二次确认
是否说明操作后果
是否有取消路径
是否有失败恢复
```

---

# 9. 交互链路审查

检查：

```text
关键操作是否可以实际触发
输入校验是否阻止无效提交并提供修正方向
处理中是否防止重复操作
成功后是否进入明确结果
失败后是否可以重试或恢复
取消、返回、撤销路径是否可用
空状态是否提供下一步
页面或步骤之间是否存在断链
资源加载或控制台错误是否阻断任务
```

不能把静态展示的多个状态视为已实现交互链路。

---

# 10. 问题优先级

## P0

影响任务完成、理解或可用性的问题。

## P1

影响效率、层级、一致性的问题。

## P2

影响细节体验、精致度的问题。

审查结果必须按优先级输出。

---

```text

原因：
修改建议：

## P1 问题

问题：
原因：
修改建议：

## P2 问题

问题：
原因：
修改建议：

## Token 检查

合规：
不合规：

## 交互链路检查

已覆盖：
缺失或断链：

## 最终建议
```

---

# 12. 禁止事项

禁止：

- 只说“更美观”
- 只说“更高级”
- 只说“层级不清”
- 不说明修改方式
- 不指出对应规则
- 不区分问题优先级
