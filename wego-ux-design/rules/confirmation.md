# 08 Requirement Confirmation Rules

> 微购 Design System Skill / rules
> Version 2.0
> 本文档定义 AI 在执行设计任务前的需求理解与确认流程。

---

# 1. 目标

AI 在进入页面类型判断和界面生成之前，必须先理解并确认六个核心维度。缺少任一维度，设计方向就可能偏离真实诉求。

阶段一不是"简单复述用户需求"，而是"需求理解与确认"。AI 必须主动补全缺失维度、识别矛盾、显式标注不确定项，确保在方向清晰的前提下才进入阶段二。

```text
用户角色 — 给谁用？能力水平、操作习惯、痛点
使用场景 — 从哪来？什么情况下用？完成后去哪？
用户任务 — 要完成什么？完成标准？
业务目标 — 成功标准是什么？只允许一个最高优先级
页面范围 — 包含什么？不包含什么？
关键业务规则 — 字段/权限/校验/状态/数量限制/风险操作
```

---

# 2. 输入分类

根据用户输入的信息密度，将需求分为四类，决定后续提问策略：

| 信号 | 类型 | 提问策略 |
|------|------|---------|
| 包含截图、Figma 链接、线框图描述、具体布局说明 | 草图型 | 少问，只补缺口 |
| 只有一句话需求、概念性描述 | 模糊型 | 逐个深挖六个维度 |
| 包含"参考""像 XX 那样""照着做"、竞品截图或链接 | 参考型 | 分析差异，确认借鉴范围 |
| 上述混合 | 混合型 | 按优先级处理：参考分析 → 草图确认 → 模糊深挖 |

---

# 3. 六个维度的确认

## 3.1 用户角色

这个页面主要给谁用？用户的能力水平、操作习惯和痛点是什么？

用户角色决定设计门槛。不知道用户角色，就会做"自以为好用但目标用户不会用"的页面。

**提问方式：**

- 草图型：复述确认。"我理解这个页面的主要用户是 [角色]，他们 [能力水平/习惯]，对吗？"
- 模糊型：追问用户画像。什么类型的用户？技术水平如何？最常犯什么错？最怕什么？
- 参考型：对比差异。"参考产品的用户是 [X]，我们的用户和它一样吗？"

**确认标准：** 能描述出具体的用户角色，不是"所有用户"。

```text
✓ "中年商家，不太熟悉手机操作，怕点错导致损失"
✓ "年轻买家，习惯快速浏览，对价格敏感"
✗ "所有用户"
✗ "C 端用户"
```

## 3.2 使用场景

用户从哪里进入这个页面？在什么情况下使用？完成后去哪里？

使用场景决定设计方式。不知道使用场景，就会做"通用但不对味"的设计。

**提问方式：**

- 草图型：问"从哪进入"和"用完之后去哪"。
- 模糊型：用 5W1H 还原场景。什么类型的用户（Who），在什么情况下（When/Where），因为什么触发（What），想做什么（How），做完去哪（Where next）。
- 参考型：对比差异。"参考产品的使用场景是 [X]，我们的场景和它一样吗？"

**确认标准：** 能描述出完整的场景链路：入口 → 使用情境 → 出口。

```text
✓ "从订单详情点击'申请售后'进入，在收到货后7天内使用，完成后回到订单详情"
✓ "从首页推荐位进入，通勤时快速浏览，看完继续刷推荐"
✗ "用户打开这个页面"
```

## 3.3 用户任务

用户来到这个页面，要完成什么事？完成的标准是什么？

这是所有设计决策的根。不知道用户任务，就会做出"功能齐全但没人用"的页面。

**提问方式：**

- 草图型：复述确认。"我理解用户在这个页面主要要完成 [任务]，完成标准是 [X]，对吗？"
- 模糊型：5Why 追问本质。不停留在表面需求，追问到用户真正的目的。
- 参考型：对比差异。"参考产品的用户任务是 [X]，我们的用户任务和它一样吗？"

**确认标准：** 能用一句话说清"用户要完成什么"，且不是功能描述而是任务描述。

```text
✓ "用户要快速找到想买的商品并加入购物车"
✓ "用户要完成退货申请并提交凭证"
✗ "做一个商品列表页"
✗ "展示订单信息"
```

## 3.4 业务目标

这个页面做成什么样算成功？最核心的一个衡量标准是什么？

业务目标决定设计方向。不知道业务目标，就会做"好看但不对"的页面。

**提问方式：**

- 草图型：直接问。"这个页面做成什么样算成功？"
- 模糊型：用选择题引导收敛。"这个页面最核心的目标是：A) 让用户快速完成操作 B) 让用户发现更多内容 C) 让用户安心完成重要操作？"
- 参考型：对比差异。"参考产品的业务目标是 [X]，我们的目标一样吗？"

**确认标准：** 能明确说出一个优先级最高的业务目标。

```text
✓ "提高下单转化率"
✓ "降低用户操作错误率"
✗ "做一个好用的页面"
✗ "既要转化率又要用户体验"
```

## 3.5 页面范围

本页面包含什么？不包含什么？

页面范围决定设计边界。不知道页面范围，就会做"什么都往里塞"或"该有的没有"的页面。

**提问方式：**

- 草图型：确认边界。"我理解本页面包含 [X]，不包含 [Y]，对吗？"
- 模糊型：主动划定范围。根据用户任务和业务目标，提出建议范围，请用户确认。
- 参考型：对比差异。"参考产品这个页面包含 [X]，我们需要一样的范围吗？"

**确认标准：** 能明确列出"本页面负责什么"和"本页面不负责什么"。

```text
✓ "包含：商品列表、筛选、排序。不包含：商品详情、购物车"
✓ "包含：退货申请表单、凭证上传。不包含：退货进度查询"
✗ "做一个商品页面"
```

## 3.6 关键业务规则

有哪些字段、权限、校验、状态、数量限制、风险操作等规则会影响设计？

业务规则决定设计约束。不知道业务规则，就会做"交互不完整"或"状态遗漏"的页面。

**提问方式：**

- 草图型：问"有没有特殊的业务规则或限制？"
- 模糊型：按类别主动追问。字段是否必填？有没有权限区分？校验规则是什么？有没有数量限制？有没有不可逆操作？
- 参考型：对比差异。"参考产品的业务规则是 [X]，我们的规则和它一样吗？"

**确认标准：** 能列出影响页面设计的关键规则，至少覆盖以下类别中与页面相关的项：

```text
- 字段规则：必填/选填、格式要求、长度限制
- 权限规则：谁可以看/操作什么
- 校验规则：提交前必须满足什么条件
- 状态规则：有哪些状态，状态间如何流转
- 数量限制：列表条数、上传数量、选择上限
- 风险操作：不可逆操作、涉及金额/权益的操作
```

# 4. Figma 集成

当用户输入中包含 Figma 链接时：

1. 检查 Figma MCP 是否可用
2. 可用 → 调用 Figma MCP 读取设计稿，提取布局结构、组件层级、交互标注，作为草图型输入处理
3. 不可用 → 提示用户："Figma MCP 未连接，请提供设计稿截图或文字描述"
4. 没有链接时跳过，不影响流程

# 5. 需求完整度判断

根据六个维度的信息完整度，判断需求是否可以进入阶段二：

| 维度状态 | 判断 | 处理方式 |
|---------|------|---------|
| 六个维度都有明确答案 | 需求完整 | 输出《需求确认卡》，等待用户确认 |
| 缺失维度影响页面方向 | 需求不完整，不可进入阶段二 | 必须向用户确认缺失维度，不得跳过 |
| 缺失维度只影响局部细节 | 需求基本完整 | 可列为"设计假设"，在确认卡中显式标注，请求用户确认 |

**影响页面方向的缺失信息（必须确认）：**
- 不知道用户角色 → 无法确定设计门槛和交互复杂度
- 不知道用户任务 → 无法确定页面核心功能
- 不知道业务目标 → 无法确定设计优先级
- 不知道页面范围 → 无法确定页面边界

**只影响局部细节的缺失信息（可列为设计假设）：**
- 具体字段是否必填 → 可假设为必填，标注设计假设
- 具体数量限制 → 可假设常见值，标注设计假设
- 具体校验规则 → 可假设基本校验，标注设计假设

# 6. 需求确认卡

六个维度确认完成后，AI 输出《需求确认卡》：

```text
📋 需求确认卡
- 我理解的用户角色：[谁，能力水平，操作习惯，痛点]
- 我理解的使用场景：[从哪来，什么情况下用，完成后去哪]
- 我理解的用户任务：[要完成什么，完成标准]
- 我理解的业务目标：[最高优先级的成功标准]
- 本次页面范围：[本页面包含什么]
- 本次不做范围：[本页面不包含什么]
- 已知业务规则：[字段/权限/校验/状态/数量限制/风险操作]
- 缺失信息与设计假设：[缺失但可假设的，标注「设计假设」]
- 需要用户确认的问题：[影响页面方向的缺失信息]
```

**输出规则：**

- 用户确认《需求确认卡》后 → 进入阶段二
- 用户修正 → 针对修正维度重新确认，更新确认卡，不需要全部重来
- 确认卡中"缺失信息与设计假设"和"需要用户确认的问题"可以为空，但字段必须保留
- 设计假设必须显式标注「设计假设」，不得混入已确认信息

---

# 7. 提问优先级

当多个维度信息缺失时，按以下优先级提问：

```text
1. 用户任务 — 不知道做什么，一切设计无从谈起
2. 业务目标 — 不知道成功标准，设计方向无法确定
3. 页面范围 — 不知道边界，设计会蔓延或缺失
4. 用户角色 — 不知道给谁用，设计门槛无法确定
5. 使用场景 — 不知道情境，设计方式可能不对味
6. 关键业务规则 — 不知道约束，设计可能不完整但方向不会错
```

每次提问不超过 2 个维度，避免用户信息过载。优先解决高优先级缺失。

---

# 8. 简短模式

当用户要求"直接输出""不要解释""给我文件"时，进入简短模式。

**简短模式不跳过阶段一，只精简输出。**

- 必须输出精简版《需求确认卡》（保留全部 9 个字段，但内容精简）
- 如果存在影响设计方向的缺失信息，最多提出 1 个关键问题
- 不得在页面方向不清晰时直接进入阶段二
- 如果用户输入已包含足够信息推断六个维度，确认卡中直接填写推断结果，不额外提问

**精简版确认卡示例：**

```text
📋 需求确认卡
- 用户角色：买家（推断）
- 使用场景：从订单列表进入，申请售后
- 用户任务：提交退货申请
- 业务目标：降低退货申请放弃率
- 页面范围：退货申请表单
- 不做范围：退货进度查询
- 业务规则：7天退货期（设计假设）
- 设计假设：凭证照片最多9张
- 待确认：无
```

# 9. 阶段一完成标准

AI 必须能明确回答以下 7 个问题，才允许进入阶段二：

| # | 问题 | 对应维度 |
|---|------|---------|
| 1 | 这是给谁用的？ | 用户角色 |
| 2 | 用户为什么进入这个页面？ | 使用场景 |
| 3 | 用户要完成什么？ | 用户任务 |
| 4 | 页面成功标准是什么？ | 业务目标 |
| 5 | 本页面负责什么？ | 页面范围 |
| 6 | 本页面不负责什么？ | 页面范围 |
| 7 | 哪些业务规则会影响设计？ | 关键业务规则 |

**不允许进入阶段二的情况：**

- 无法回答问题 1-4 中的任何一个 → 页面方向不清晰
- 页面范围模糊到无法区分"做什么"和"不做什么" → 边界不清晰
- 存在影响页面方向的缺失信息且未向用户确认 → 信息不充分

**允许进入阶段二的情况：**

- 7 个问题都有明确答案（含设计假设）
- 影响页面方向的信息已确认
- 缺失信息已列为设计假设并在确认卡中标注
- 用户已确认《需求确认卡》
