# 08 Requirement Confirmation Rules

> 微购 Design System Skill / rules
> Version 3.0
> 本文档定义 AI 在执行设计任务前的需求确认门禁。

---

# 1. 目标

阶段一不是复述需求，而是确认页面方向。进入阶段二前，AI 必须明确以下六个维度：

```text
用户角色 — 给谁用？能力水平、操作习惯、痛点
使用场景 — 从哪来？什么情况下用？完成后去哪？
用户任务 — 要完成什么？完成标准？
业务目标 — 成功标准是什么？只允许一个最高优先级
页面范围 — 包含什么？不包含什么？
关键业务规则 — 字段/权限/校验/状态/数量限制/风险操作
```

---

# 2. 确认规则

- 优先从用户输入直接提炼六个维度，不做无意义复述。
- 缺失信息影响页面方向时，必须向用户确认，不得进入阶段二。
- 任何不清楚的信息都必须向用户确认，不得自行补成「设计假设」。
- 每次只追问最关键的缺口，避免把阶段一变成长访谈。

以下信息缺失时，视为影响页面方向：

- 用户任务
- 业务目标
- 页面范围
- 用户角色

提问优先级：

```text
1. 用户任务
2. 业务目标
3. 页面范围
4. 用户角色
5. 使用场景
6. 关键业务规则
```

---

# 3. 需求确认卡

本节是《需求确认卡》的唯一权威描述。其他运行时文件只能引用本节，不得重复定义另一份模板。

六个维度确认完成后，AI 输出《需求确认卡》：

```text
📋 需求确认卡
- 用户角色：[谁，能力水平，操作习惯，痛点]
- 使用场景：[从哪来，什么情况下用，完成后去哪]
- 用户任务：[要完成什么，完成标准]
- 业务目标：[最高优先级的成功标准]
- 页面范围：[本页面包含什么]
- 不做范围：[本页面不包含什么]
- 业务规则：[字段/权限/校验/状态/数量限制/风险操作]
- 待确认：[仍不明确、必须由用户确认的信息]
```

输出约束：

- 生成类任务的首轮回复只能输出《需求确认卡》
- 《需求确认卡》整体必须放在单个 `text` 代码块中输出，不得改写为普通段落、普通列表或表格
- 用户确认《需求确认卡》后 → 进入阶段二
- 用户修正 → 只更新受影响字段，不需要全部重来
- 确认卡中"待确认"可以为空，但字段必须保留
- 字段名必须严格使用本节短标签；不得改写为"我理解的用户角色""缺失信息与设计假设""需要用户确认的问题"等旧写法
- 未获用户确认前，不得输出原型代码、项目目录、文件内容、组件方案、页面方案、交互实现或在线链接

---

# 4. 阶段一完成标准

只有满足以下条件，才允许进入阶段二：

- AI 能回答：给谁用、为何进入、完成什么、成功标准、负责范围、不做范围、影响设计的业务规则
- 影响页面方向的缺失信息已确认
- 用户已确认《需求确认卡》

以下情况不允许进入阶段二：

- 无法明确用户任务、业务目标、页面范围或用户角色
- 页面范围模糊到无法区分“做什么”和“不做什么”
- 存在影响页面方向的缺失信息且未向用户确认

---

# 5. Figma

当用户输入包含 Figma 链接时：

- Figma MCP 可用：读取设计稿后再执行本规则
- Figma MCP 不可用：提示用户提供截图或文字描述
