# Checkout Rules

> 微购 Design System Skill / rules
> Version 3.0
> 本文档定义 AI 输出前必须执行的最终检查清单。对照阶段二设计产物验证，而非泛泛检查。

---

# 1. Step 1：验证

**方式 A：浏览器验证（环境支持时）**

```text
□ 关键交互链路可操作（对照阶段二 2.6 状态链）
□ 控制台无阻断错误
□ 资源加载成功（字体、图标、图片）
□ 响应式表现合理（375px 基准 + 宽屏验证）
```

**方式 B：静态代码审查（环境不支持浏览器时）**

```text
□ 资源路径在交付目录内可解析（字体、图标、图片文件存在）
□ CSS 中所有颜色、间距、字号、圆角通过 var(--wg-*) 引用
□ JavaScript 中状态转换逻辑完整覆盖阶段二 2.6 状态链
□ HTML 语义结构正确，无内联业务 CSS 和 JavaScript
□ 样式加载顺序正确：tokens.css → iconfont.css → components.css → app.css
```

---

# 2. Step 2：设计产物落地验证

对照阶段二的设计决策产物逐项验证是否落地。每项检查引用产物编号，有明确对照对象。

## 2.1 页面目标与类型

```text
□ 页面类型对应的布局特征是否体现
□ 页面目标是否通过主操作和内容层级传达
```

## 2.2 信息层级

```text
□ 一级核心视觉重心是否只有一个
□ 二级主要内容是否清晰
□ 三级辅助信息是否弱化
□ 状态信息是否靠近对应对象
□ 同一区域是否不超过三级文字层级
```

## 2.3 布局模式

```text
□ M[0-3] 页面留白是否使用定义选项
□ G[1-2] 分组是否使用定义选项
□ 页面边距是否稳定
□ 模块间距是否合理
□ 是否没有临时 padding 伪造布局
□ 页面背景色选择是否符合内容结构（多卡片 bg-page / 聚焦 bg-surface）
□ 页面高度是否使用 100vh/100dvh，滚动策略是否正确
□ 滚动区域是否隐藏浏览器默认滚动条
□ 页面转场是否使用 Push/Present/Fade 动画而非硬切
□ 页面主体最大宽度是否 ≤ --wg-layout-page-max-width
□ 大屏下页面是否没有被过度拉伸
□ 滚动后导航区是否仍可见
□ 固定区 z-index 是否正确（使用 wg.zIndex.* Token）
□ 底部操作区在长内容页面是否固定
□ 是否完成页面结构模式判断
□ 是否明确顶部固定区、主体滚动区、底部固定区
□ Search / Filter / Tabs 是否按场景固定
□ ScrollView 是否为底部固定栏预留安全距离
□ 卡片是否对应真实业务实体
□ 简单设置项 / 表单项是否没有滥用卡片
□ 多列表格是否支持横向滚动
□ 批量操作页是否有固定底部选择工具栏
```

## 2.4 组件匹配

```text
□ 命中组件是否通过 `design-library/components/index.json` 选择
□ 命中组件是否遵守对应组件契约（Anatomy / 状态 / 可访问性）
□ 是否先判断组件语义维度，再选择视觉变体
□ 是否没有自创组件变体
□ 未注册结构是否使用页面级命名并标注为组件候选
□ 是否没有读取或引用 KuiklyUI 组件 API
□ 命中组件的 Canonical CSS 是否逐字复制写入 styles/components.css（未增删规则、未修改选择器、未添加属性）
□ 页面中所有可点击的 `<a>` 标签是否使用了 `.wg-link` 组件（包括协议链接、帮助链接、注册入口等）
□ 可点击链接的 pressed / hover / focus-visible 状态是否符合 Link 组件规范（不得用 `opacity` 替代 Link 的 pressed 态背景色）
□ page 级按钮宽度是否符合宽度策略（表单型用 `--form` 修饰符 / 结果型用固定宽度）
□ 按钮宽度是否通过 Button Group 修饰符控制（未在 app.css 中直接覆盖 `.wg-button` 的 `inline-size`）
□ NavBar 背景色是否与页面背景色联动（白底页面 NavBar 白色）
□ 是否先组织页面，再选择组件
□ 是否存在组件堆叠、缺少排版设计的问题
□ 同一业务动作是否只有一个 primary / strong
□ 重复业务动作是否已降级为 text / weak / secondary
□ 同一个 form group 内对齐方式是否一致
□ 错误修正后是否清除 error 状态
□ Loading 是否表达具体任务，不是空转圈
□ Dialog 内容间距、文本限制、按钮点击态是否完整
□ 所有关键组件是否覆盖 default / hover / active / disabled / error / loading 状态
□ 状态链路是否覆盖初始、输入、校验、处理中、成功、失败、恢复
```

## 2.5 Token 类别

```text
□ 阶段二 2.5 确定的 Token 类别是否全部通过 var(--wg-*) 使用
□ 是否使用 wg.color.*
□ 是否使用定义内字号
□ 是否使用定义内间距
□ 是否使用定义内圆角
□ 是否使用定义内尺寸
□ 是否使用 wg.zIndex.*
□ Token 定义层之外是否没有 HEX / RGBA 硬编码
□ 业务样式是否没有魔法数字
```

## 2.6 交互链路

```text
□ 已注册组件是否遵守其状态契约
□ 是否使用原生属性或 ARIA 表达状态
□ 状态链是否完整实现（初始、处理中、成功、失败、恢复）
□ 关键状态是否通过真实交互转换，而不是静态并列展示
□ 不可逆操作是否有确认文案、后果说明、取消路径、失败恢复
```

---

# 3. Step 3：合规验证

```text
□ 业务 CSS 无硬编码设计值
□ 未自创正式组件变体
□ 无装饰性视觉（复杂渐变、重阴影、多层卡片）
□ HTML 不内联业务 CSS
□ HTML 不内联业务 JavaScript
□ 文案清晰，操作文案使用动词
□ 空状态说明当前状态
□ 错误状态有恢复路径
□ 危险操作说明后果
□ 避免恐吓式表达
□ 视觉风格简洁、干净、克制、淡雅、符合微信生态
□ 是否完成页面目标判断
□ 是否完成信息结构判断
□ 是否完成页面组合判断
□ 是否完成操作层级判断
□ 组件 CSS 是否不存在非法单位
```

---

# 4. 最终交付检查

```text
□ 结果是否可执行
□ 结构是否清晰
□ 是否符合用户要求
□ 是否交付完整项目目录
□ HTML、CSS、JavaScript 是否分离
□ 本地资源路径是否有效
□ 是否包含入口和启动方式
□ 是否实际验证关键交互链路
□ 浏览器控制台是否无阻断错误
□ 是否没有输出推理过程
□ 是否没有多余解释
□ 是否可直接交付开发
□ 项目根目录名是否使用中文、精简表达页面内容（禁止 prototype/demo/page1 等通用占位名）
□ iconfont 资源目录结构是否完整（iconfont.css 与字体文件在同一目录）
```

---

# 5. 门禁规则

- 自查清单逐项通过后才允许输出
- 未通过项必须修复后重新验证
- 不得跳过任何步骤直接输出
- 不得在未完成阶段二设计产物时进入验证
- 如果任一关键项不满足，必须重新修改，不得交付
